WHY THIS EXISTS
---------------
dump_articles.ts dumps one JSON per article with a "metadata" object and a
"content" object. For 5 of the 15 document types the Coveo SEARCH INDEX returns
no full body field, so their content is empty (content: []):

    Type                    7-day count   total corpus   body source needed
    Manual                  3,338         46,811         scrape doc HTML pages
    Bug Tracker             3             22,245         scrape cdn.f5.com bug HTML
    F5 GitHub               6              1,234          GitHub REST API
    Release Note            35            757            scrape doc HTML pages
    Supplemental Document   20            135            scrape doc HTML pages

The real body lives on the public rendered page that each article's `link`
(clickableuri) points to — NOT in the search API. Goal: fetch those pages,
extract the main body, and populate the `content` object (the metadata is
already complete and correct for all 15 types).

All target pages were spot-checked on 2026-06-02 and return HTTP 200 publicly
(no login). Approaches below are validated, not speculative.

RECOMMENDED IMPLEMENTATION
--------------------------
Build a SEPARATE enrichment script — proposed name: enrich_bodies.ts — that
post-processes an existing dump directory rather than bloating dump_articles.ts
(body scraping has very different concerns: HTML parsing, per-host rules, rate
limiting, robots/politeness, resumability).

  deno run --allow-net --allow-read --allow-write enrich_bodies.ts \
      --dump=dump [--types="Bug_Tracker,Manual"] [--concurrency=4] [--limit=N]

Behavior:
  - Walk dump/<Type>/*.json for the configured empty-body types.
  - For each, derive the body URL (or API call) from existing fields.
  - Fetch, extract main content, convert HTML -> text/markdown.
  - Write the result back into the article JSON's `content` object
    (e.g. content.body_html / content.body_text) and record content.bodySource
    + content.fetchedAt. Make it RESUMABLE: skip articles that already have a
    non-empty content.body unless --refetch is passed.
  - Reuse the existing coveoPost-style retry/backoff pattern for transient
    network errors (already added to dump_articles.ts — copy that logic).

Likely new dependency: an HTML parser. Use deno-dom or linkedom from JSR/esm
for selector-based extraction; optionally a html->markdown converter
(turndown / html-to-text). Confirm import works under Deno 2.8.

A small companion config (proposed: dump_bodies_config.yaml) mapping
host -> CSS selector keeps per-site extraction rules out of the code.

----------------------------------------------------------------------
CROSS-CUTTING / RISKS
----------------------------------------------------------------------
- Resumability is essential (70k+ total fetches). Skip already-enriched
  articles; support --refetch.
- Politeness: check each host's robots.txt; set a descriptive User-Agent; cap
  concurrency; add backoff on 429/5xx (reuse dump_articles.ts coveoPost retry).
- Decide storage: inline into the per-article JSON (content.body_html +
  content.body_text/markdown) vs. sidecar files. Inline is simplest and keeps
  one-file-per-article; large Manual bodies may bloat files — acceptable.
- Idempotence: enrich_bodies.ts should be safe to re-run after a fresh
  dump_articles.ts run.
- Validation after build: spot-check a few enriched articles per type; confirm
  body_text is non-empty and looks like the real page content (not nav/boiler-
  plate); report per-type coverage (how many got a body vs. 404/empty).
