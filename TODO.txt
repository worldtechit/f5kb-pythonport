DONE: CONSOLIDATION INTO ONE CLI
--------------------------------

single `f5kb` CLI with subcommands plus a `lib/` module tree. Everything is now:

is byte-identical, and `f5kb track` over the 106k dump reported 0 new / 0 changed).

DONE 2026-06-04: INCREMENTAL SYNC + DELETION RECONCILE + CHANGELOG
-----------------------------------------------------------------
The deferred "skip unchanged bodies" idea shipped as `f5kb sync`, plus deletion
handling and a changelog. See README.md (sync/reconcile + CHANGELOG FORMAT) and
OUTLINE.txt §2-3.
  - `f5kb sync` loads each article's prior metadata_hash from the DB and SKIPS
    rewriting unchanged articles; enrich's existing resumability then auto-limits
    body fetches to the rewritten (changed) files — so unchanged bodies are never
    re-scraped. Modes: --all, --days=N, --since-last-run. Detection of upstream
    deletions runs under --all only (needs the full live id set) and is REPORTED,
    never executed.
  - `f5kb reconcile` is the explicit, guarded executor for deletions (report-only
    unless --apply; --max-delete-pct / --max-deletes guard; DB backup; soft-delete
    to _deleted/<type>/ or --purge). It is the only command that removes data.
  - `--changelog[=FILE]` on every mutating op (sync/dump/enrich/track/reconcile)
    appends JSONL change records; sync writes one by default.
  - Latent-bug guard: dbKey() must match loadHashIndex()'s "<document_type> <id>"
    key byte-for-byte or skip-unchanged silently no-ops; locked by
    test/integration/sync_cmd_test.ts (a real DB round-trip).

Possible follow-ups (not now):
  - body-only upstream changes that bump no date are not detected by metadata_hash
    (rare); `enrich --refetch` remains the escape hatch.
  - a combined "sync then reconcile --apply" wrapper, if the report-then-execute
    two-step ever proves tedious in practice.


DONE 2026-06-04: OVERWRITE PROTECTION (APPROVAL GATE) + approve COMMAND
----------------------------------------------------------------------
sync/dump/enrich no longer silently overwrite a live article that already holds
good data (the "upstream reformats and we capture an empty body" risk). See
README.md "OVERWRITE PROTECTION" + "SUBCOMMAND — approve", OUTLINE.txt §2-3/§6,
and HOWTO.txt.
  - An EDIT to an existing article is STAGED to <dump>/_pending/<type>/<id>.json
    (live untouched) and recorded in _pending/_manifest.json; new articles are
    written directly, unchanged are skipped.
  - `f5kb approve` promotes staged edits: archive the replaced live file to
    <dump>/_replaced/<type>/<id>.<ts>.json, move pending -> live, reindex the DB.
    Edits flagged risky (body-dropped / body-error, recomputed fresh from the files)
    are HELD unless --include-risky; --list previews, --reject discards.
  - `--yes` on sync/dump/enrich bypasses the gate (overwrite in place, still
    archiving to _replaced/).
  - listTypeDirs now skips _-prefixed dirs so _pending/_replaced/_deleted are never
    indexed as article types. Tests: +9 (staging, approve, gate) -> 125 green.

Idea considered, NOT built: gating a full `dump` of 100k+ articles can stage a huge
_pending/ — acceptable because a first dump (empty dir) is all-new (nothing staged)
and re-dumps over an existing corpus only stage genuine edits. If mass-staging ever
bites, add a `--max-staged` guard that aborts like reconcile's threshold.


DONE 2026-06-04: --exclude-types EVERYWHERE + README -> README.md
----------------------------------------------------------------
Every type-aware subcommand (dump/enrich/track/sync/reconcile/approve/recent) now
takes `--exclude-types=A,B` alongside the existing `--types=A,B`. Include is applied
first, then exclude (exclude wins on a conflict); unknown keys are warned + ignored.
Resolution is the shared `applyTypeFilters` / `warnUnknownTypes` in lib/args.ts
(cmd-side for the config-driven commands; `excludeTypes` params for enrich/track and
`excludeTypeKeys` for approve). The canonical production selection is
`--exclude-types=Community,F5_GitHub`. Tests: +5 (applyTypeFilters, warnUnknownTypes,
track/enrich/approve exclude) -> 130 green.

README.txt was converted to README.md (proper Markdown; GitHub-renderable) and is
the doc format going forward; all cross-references updated. config.yaml is excluded
from `deno fmt` (it reformats YAML) so the curated file isn't rewritten. The other
docs (FINDINGS/OUTLINE/HOWTO/TODO) remain .txt per CLAUDE.md's conventions.
